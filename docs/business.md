# 工业设备通信网关商业化说明

更新时间：2026-05-14

本文档用于统一团队对 `EnviroEquipmentFinalEdition` 的商业化判断：这是独立第二业务线，定位为工业设备通信网关、PLC 数据采集、现场诊断、老系统升级的项目制现金流业务，不并入教育品牌，不使用教育业务的品牌表达、客户承诺和销售叙事。

## 项目定位

`EnviroEquipmentFinalEdition` 当前应定位为面向工业现场的设备通信与诊断交付项目，而不是单纯的技术 demo 或旧 Qt 工程修补。

商业定位：

- 独立第二业务线：面向工厂、设备商、系统集成商、实验室设备和非标自动化现场。
- 项目制现金流：用现场诊断、PLC 数据采集、老系统升级、运行维护形成可收款项目。
- 通信内核优先：先把 S7、Modbus、点位配置、轮询、诊断、日志做成可复用核心，再按客户项目接 UI、服务、报表或上位系统。
- 现场交付优先：卖点不是概念平台，而是能到现场把 PLC 连上、点读出来、问题定位清楚、旧系统逐步升级。

当前仓库的技术路线已经贴近这个定位：以 C# / .NET 8 为通信内核，保留命令行 runner、诊断脚本和项目配置文件，后续可以扩展为 Windows Service、安装包、本地数据库和上层界面。

## 目标客户

优先客户画像：

- 中小制造现场：有 PLC 设备，但缺统一数据采集、运行记录和故障诊断能力。
- 设备厂商：已有设备或上位机程序，需要补一套可交付的数据采集、远程排障或新版本升级能力。
- 系统集成商：接到改造项目，需要快速打通 Siemens S7、Schneider / Modbus 等设备通信。
- 实验室与环境试验设备现场：设备分散、点位表老旧，需要把温度、压力、状态、报警等数据统一采集。
- 老系统维护团队：旧 Qt / XML / 点位配置仍能用，但难维护、难扩展、难交付给新客户。

第一批客户不追求大平台销售，优先选择能接受项目制试点、现场设备明确、付款链路短、验收标准清楚的客户。

## 核心痛点

客户现场通常不是缺一个页面，而是缺一套能落地的通信和诊断能力：

- PLC 能不能连上不清楚：IP、网卡、VPN/TAP 路由、TCP 端口、rack/slot、connection type 混在一起排查。
- 点位能不能读不清楚：地址、数据类型、缩放、字节序、单位、权限经常散落在旧 XML、PLC 工程或人工表格里。
- 老系统难升级：旧程序能跑但不好扩展，新增协议、设备、日志和自动化验收成本高。
- 现场问题难复盘：只有控制台输出或人工截图，没有结构化日志和可重复的诊断脚本。
- 集成商交付压力大：每个现场都从头试协议、写脚本、修点位，难形成标准报价和可复用交付件。

本业务线要解决的是“把设备通信这件事产品化交付”：用固定诊断流程、配置模板、协议适配器、运行日志和验收标准，把现场不确定性转化为可报价、可验收、可复用的工程交付。

## 付费产品与收费方式

建议先采用项目制和服务包，不急于包装成 SaaS 订阅。

| 付费产品 | 交付内容 | 收费方式 |
|---|---|---|
| 现场通信诊断包 | 网卡/路由/TCP/协议握手/点位读取诊断，输出诊断记录和下一步改造清单。 | 按现场或按天收固定诊断费；后续签正式项目时可抵扣一部分。 |
| PLC 数据采集网关交付包 | 按客户设备和点位表配置 S7 / Modbus 采集，交付 runner、配置、日志、操作说明和验收记录。 | 按项目报价，维度包括设备数、协议数、点位数、现场天数、是否需要服务化部署。 |
| 老系统点位迁移包 | 从旧 XML / 旧上位机配置导入设备和点位，转换为新项目 JSON，完成只读冒烟和配置校验。 | 按旧项目数量、点位规模和协议类型收费。 |
| 现场运行维护包 | 定期检查网关运行状态、处理点位变更、补充诊断报告、协助现场排障。 | 按月、季度或年度服务费。 |
| 定制集成包 | 对接客户 MES、SCADA、数据库、报表、Web/API 或新 UI。 | 单独项目报价，按接口数量、部署环境和验收范围拆分里程碑。 |

推荐收款节奏：

- 诊断包：启动前一次性收款。
- 试点项目：启动款、现场演示款、验收尾款三段式。
- 正式项目：按设备组、点位导入、现场部署、连续运行验收拆里程碑。
- 运维包：预付制，避免把现场长期排障变成免费支持。

## 当前可演示功能

当前仓库已经具备可以对内和对试点客户演示的基础能力：

- Siemens S7-200 SMART 真实连接已验证：`192.168.2.180:102`，物理网卡 `192.168.2.10/24`，Snap7 `connection-type=basic`、`rack=0`、`slot=0`。
- 一键网络与握手诊断：`tools\connect-plc.ps1`、`tools\test-plc-network.ps1` 可以先确认路由、TCP 端口和 Snap7 握手。
- 只读点位读取已验证：默认 S7-200 SMART 示例点位 `V0.0`、`VW0`、`VW2`、`VD4`、`VD8`、`M0.0` 返回 GOOD。
- 项目 JSON 模式已可运行：`src\SiemensS7Demo\Config\project.sample.json` 能按项目配置连接并读取设备。
- 配置校验和自测已可运行：`--validate-config`、`--self-test` 覆盖默认 S7 XML、项目 JSON、mock 写保护和 Modbus loopback。
- 有限轮询可验收：`--cycles` 可以跑固定轮次，适合演示和自动化验收。
- 结构化运行记录已落地：`--run-log` 可输出 JSONL snapshot，不只依赖控制台输出。
- 旧项目点位导入已验证：`tools\import-legacy-config.ps1` 可从旧项目导入配置，已生成旧 S7 冒烟配置并完成 30 个 S7 点位真实只读 GOOD。
- Snap7 依赖已项目内置：`src\SiemensS7Demo\Native\Snap7\win64\snap7.dll` 已随项目管理，不依赖旁边的 `snap7` checkout。
- 写入保护框架已存在：真实写入必须同时满足命令行开关和点位配置；当前真实 PLC 写入尚未作为演示卖点，先以只读采集和诊断成交。
- Modbus TCP 已有代码与本地 loopback 验证：适合作为下一步现场试点内容，但真实 Modbus 设备兼容性需要在客户现场确认。

对外演示建议先聚焦三件事：一键诊断能看出走哪块网卡、S7 能握手、点位能读并落日志。不要先展示复杂 UI。

## 下一个付费试点

建议下一个付费试点定义为“单现场 PLC 只读数据采集与诊断网关试点”。

试点目标：

- 在一个真实现场交付可运行的采集网关原型。
- 以一台 Siemens S7-200 SMART 或同类 S7 设备为主目标，优先做只读采集。
- 从客户旧点位表或现有上位机配置导入一批真实业务点位。
- 输出可复用交付件：项目 JSON、点位表、运行日志、诊断报告、启动脚本、验收记录。

试点范围：

- 设备数量：先限定 1 到 2 台 PLC。
- 协议范围：S7 作为主线；如现场有 Modbus 设备，作为第二阶段验证项。
- 点位范围：先从 30 个只读点扩展到客户认可的一批关键点位，再逐步覆盖完整表。
- 运行范围：先完成单次读取、有限轮询、结构化日志，再补连续运行和服务化部署。

试点验收口径：

- 现场网络诊断可明确输出网卡、源地址、TCP 连通和协议握手结果。
- 关键只读点位返回 GOOD，并能说明地址、类型、单位、缩放和业务含义。
- 每轮采集结果能落 JSONL 日志，后续可导入数据库或客户系统。
- 交付物能由团队成员按文档重复运行，不依赖某一个开发者手工操作。

试点收费建议：

- 先收诊断启动费，覆盖到现场确认网络、协议和点位表的成本。
- 通过连接和首批点位读取后，转为试点交付费。
- 试点验收后，按完整点位覆盖、Windows Service、SQLite、上层界面或客户系统对接追加正式项目报价。

## 30/60/90 天落地路线

| 时间 | 目标 | 主要动作 | 可交付物 |
|---|---|---|---|
| 30 天 | 把当前 demo 打磨成可售试点包 | 固化 S7 诊断流程；整理演示脚本；完善旧点位导入；补关键点位字段；形成报价模板和验收清单。 | 试点报价单、现场诊断清单、S7 演示包、项目 JSON 模板、点位导入说明。 |
| 60 天 | 完成第一个付费试点 | 进入真实现场；扩大 S7 点位读取数量；验证一台真实 Modbus 设备或明确排除；补运行日志归档；开始 Windows Service / SQLite 原型。 | 试点验收报告、客户现场配置包、运行日志样例、问题闭环记录、下一阶段正式项目方案。 |
| 90 天 | 形成可复制交付版本 | 抽象 DeviceGateway 核心结构；补多设备状态、断线重连、降频重试；形成安装/启动/排障标准流程；准备第二个试点或正式项目交付。 | 可复用网关版本、安装包或服务化部署脚本、标准操作手册、第二客户复制方案、运维服务报价。 |

路线执行原则：

- 先卖能跑的诊断和采集，不先卖完整平台。
- 先做只读采集和日志，不把真实写入作为第一笔收入前提。
- 先服务少量明确现场，沉淀模板后再扩大客户类型。
- 每个试点都要沉淀成配置、脚本、日志和验收记录，避免一次性外包式交付。

## 关键指标

业务指标：

- 已收费诊断现场数量。
- 诊断包转试点项目比例。
- 试点项目平均回款周期。
- 单项目毛利和现场人天。
- 旧系统升级项目复购或追加模块金额。
- 运维包签约数量和续费率。

交付指标：

- 从进场到完成 `connect-only` 的时间。
- 从拿到点位表到首批 GOOD 点位的时间。
- 每个项目可复用配置和脚本比例。
- 现场问题定位是否能区分 route、tcp、protocol、tag、config。
- 团队成员按文档独立复现演示的成功率。

技术指标：

- 设备连接成功率。
- GOOD 点位比例。
- 连续轮询运行时长。
- BAD 点位严格失败覆盖率。
- 结构化日志完整率。
- 旧点位导入成功数量和需人工修正数量。
- S7、Modbus 两条协议的真实现场验收数量。
